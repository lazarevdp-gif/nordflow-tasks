# Техническое задание на разработку веб-приложения NordFlow Tasks

## 1. Информация о заказчике

**Компания:** NordFlow
**Сфера деятельности:** разработка цифровых продуктов и автоматизация бизнеса
**Количество сотрудников:** около 35 человек
**Контактное лицо:** Дмитрий Лазарев, операционный директор

## 2. Название продукта

**NordFlow Tasks**

Рабочее название может использоваться в интерфейсе, заголовке вкладки браузера и на странице авторизации.

## 3. Цель проекта

Разработать внутренний трекер проектов и задач, который позволит сотрудникам компании:

* хранить все рабочие задачи в одном месте;
* видеть текущий статус проектов;
* распределять задачи между сотрудниками;
* контролировать сроки;
* обсуждать задачи внутри сервиса;
* отслеживать загрузку команды.

Приложение должно быть проще Jira и не требовать длительного обучения сотрудников.

## 4. Формат продукта

Необходимо создать адаптивное веб-приложение, которым можно пользоваться:

* на компьютере;
* на планшете;
* на смартфоне.

На компьютере должна отображаться полная версия интерфейса. На смартфоне элементы должны перестраиваться под вертикальный экран, не выходить за его границы и оставаться удобными для нажатия.

## 5. Пользователи и роли

### 5.1. Администратор

Может:

* просматривать все проекты;
* создавать и редактировать проекты;
* добавлять и удалять пользователей;
* назначать руководителей проектов;
* просматривать все задачи;
* менять роли пользователей;
* архивировать проекты.

### 5.2. Руководитель проекта

Может:

* создавать задачи внутри своих проектов;
* редактировать задачи;
* назначать исполнителей;
* менять сроки и приоритеты;
* просматривать задачи всей проектной команды;
* перемещать задачи между статусами;
* добавлять участников в проект.

### 5.3. Сотрудник

Может:

* просматривать проекты, в которые он добавлен;
* видеть назначенные ему задачи;
* менять статус своих задач;
* оставлять комментарии;
* создавать задачи, если это разрешено внутри проекта;
* прикреплять ссылки и файлы.

## 6. Основные разделы приложения

После авторизации пользователь попадает в основную рабочую область.

В приложении должны быть следующие разделы:

1. Главная.
2. Мои задачи.
3. Проекты.
4. Команда.
5. Уведомления.
6. Настройки профиля.

Для администратора дополнительно должен отображаться раздел управления пользователями.

## 7. Авторизация

Необходимо реализовать страницу входа.

Поля:

* электронная почта;
* пароль;
* кнопка «Войти».

Дополнительно:

* возможность показать или скрыть пароль;
* сообщение при неправильном логине или пароле;
* кнопка выхода из аккаунта;
* сохранение авторизации после обновления страницы.

Самостоятельная регистрация сотрудников в MVP не требуется. Пользователей создаёт администратор.

## 8. Главная страница

На главной странице пользователь должен видеть краткую информацию о своей работе.

Необходимо вывести:

* количество активных задач;
* количество просроченных задач;
* количество задач, которые нужно завершить на текущей неделе;
* список ближайших дедлайнов;
* последние обновления в проектах;
* блок «Мои проекты».

Для руководителя проекта дополнительно показать краткую сводку по задачам команды.

## 9. Раздел «Мои задачи»

В разделе отображаются все задачи, где текущий пользователь назначен исполнителем.

Для каждой задачи показать:

* название;
* проект;
* статус;
* приоритет;
* срок выполнения;
* имя постановщика.

Необходимо добавить фильтры:

* по статусу;
* по проекту;
* по приоритету;
* по сроку;
* только просроченные задачи.

Также необходим поиск по названию задачи.

Пользователь должен иметь возможность выбрать вид отображения:

* список;
* доска по статусам.

## 10. Раздел «Проекты»

На странице проектов отображаются карточки доступных пользователю проектов.

Карточка проекта содержит:

* название;
* краткое описание;
* руководителя;
* количество открытых задач;
* количество участников;
* дату завершения проекта;
* текущий статус.

Статусы проекта:

* планируется;
* активный;
* приостановлен;
* завершён;
* архивный.

На странице должна быть кнопка «Создать проект», доступная администратору.

## 11. Создание проекта

При создании проекта необходимо заполнить:

* название проекта;
* описание;
* руководитель проекта;
* участники;
* дата начала;
* плановая дата завершения;
* статус.

После сохранения проект появляется в общем списке и становится доступен его участникам.

## 12. Страница проекта

Внутри проекта необходимо реализовать вкладки:

* «Доска»;
* «Список задач»;
* «Участники»;
* «Информация о проекте».

### Вкладка «Доска»

Задачи отображаются в виде Kanban-доски.

Колонки:

1. Бэклог.
2. Запланировано.
3. В работе.
4. На проверке.
5. Выполнено.

Пользователь должен иметь возможность перетаскивать карточки задач между колонками. После перемещения статус задачи должен автоматически изменяться.

На смартфоне колонки могут отображаться горизонтально с прокруткой либо переключаться с помощью списка статусов.

## 13. Карточка задачи

На карточке задачи необходимо показать:

* название;
* идентификатор задачи, например NDF-125;
* исполнителя;
* статус;
* приоритет;
* дедлайн;
* количество комментариев;
* отметку о наличии вложений.

Цвет или визуальная метка приоритета должны отличаться:

* низкий;
* средний;
* высокий;
* критический.

Просроченные задачи необходимо визуально выделять.

## 14. Создание и редактирование задачи

Форма задачи должна содержать следующие поля:

* название;
* подробное описание;
* проект;
* статус;
* исполнитель;
* постановщик;
* приоритет;
* дата создания;
* дедлайн;
* теги;
* прикреплённые файлы или ссылки.

Обязательные поля:

* название;
* проект;
* исполнитель;
* статус;
* приоритет.

После сохранения задача должна появляться на доске выбранного проекта.

## 15. Страница задачи

При открытии задачи пользователь должен видеть всю информацию о ней.

На странице необходимо разместить:

* название и идентификатор;
* описание;
* проект;
* текущий статус;
* приоритет;
* исполнителя;
* постановщика;
* дату создания;
* дедлайн;
* теги;
* вложения;
* историю изменений;
* комментарии.

Поля задачи должны редактироваться без перехода на отдельную страницу либо через кнопку «Редактировать».

## 16. Комментарии

В каждой задаче пользователи могут оставлять комментарии.

Для комментария показать:

* имя и фотографию автора;
* текст;
* дату и время публикации.

Пользователь может удалять или редактировать только собственные комментарии. Администратор может удалять любые комментарии.

В первой версии достаточно текстовых комментариев без сложного форматирования.

## 17. Уведомления

В верхней части интерфейса необходимо разместить иконку уведомлений.

Уведомление создаётся, если:

* пользователю назначили новую задачу;
* изменился срок его задачи;
* изменился статус задачи;
* в его задаче появился новый комментарий;
* приближается дедлайн;
* задача стала просроченной.

В MVP уведомления должны отображаться только внутри приложения. Отправка на электронную почту и в Telegram будет реализована позднее.

## 18. Раздел «Команда»

В разделе отображается список сотрудников.

Для каждого сотрудника показать:

* имя и фамилию;
* должность;
* фотографию или инициалы;
* электронную почту;
* количество активных задач;
* количество просроченных задач;
* текущую загрузку.

При открытии карточки сотрудника руководитель может посмотреть его активные задачи.

## 19. Профиль пользователя

В профиле пользователь может:

* изменить имя и фамилию;
* указать должность;
* загрузить фотографию;
* изменить пароль;
* выбрать светлую или тёмную тему интерфейса.

Электронную почту и роль может изменять только администратор.

## 20. Поиск

В верхней части приложения должен быть глобальный поиск.

Поиск должен находить:

* проекты;
* задачи;
* сотрудников.

Результаты должны группироваться по типу объекта.

## 21. Адаптивность

Приложение должно корректно работать на экранах разной ширины.

### На компьютере

* постоянное боковое меню;
* несколько колонок на Kanban-доске;
* таблицы и списки во всю ширину;
* карточка задачи может открываться в боковой панели или отдельном окне.

### На смартфоне

* боковое меню заменяется кнопкой-гамбургером;
* карточки располагаются в одну колонку;
* формы открываются на весь экран;
* таблицы превращаются в карточки;
* кнопки имеют удобный размер;
* основные действия доступны без горизонтальной прокрутки.



## 23. Данные для демонстрации

В демонстрационной версии необходимо создать:

* 8 пользователей;
* 3 проекта;
* не менее 25 задач;
* задачи с разными статусами и приоритетами;
* несколько просроченных задач;
* комментарии и историю изменений.

Примеры проектов:

1. Разработка сайта для клиента «GreenStone».
2. Запуск рекламной кампании NordFlow.
3. Внутренняя автоматизация отдела продаж.

## 24. Основные пользовательские сценарии

### Сценарий 1. Создание задачи

Руководитель открывает проект, нажимает «Создать задачу», заполняет форму, назначает исполнителя и устанавливает дедлайн. Задача появляется на доске, а исполнитель получает уведомление.

### Сценарий 2. Работа сотрудника с задачей

Сотрудник открывает раздел «Мои задачи», выбирает задачу, читает описание и переводит её в статус «В работе». После выполнения переводит задачу в статус «На проверке» и оставляет комментарий.

### Сценарий 3. Контроль проекта

Руководитель открывает доску проекта, видит распределение задач по статусам, просроченные задачи и загрузку участников. При необходимости меняет исполнителя или срок.

### Сценарий 4. Работа со смартфона

Сотрудник открывает приложение на смартфоне, просматривает свои задачи, меняет статус и оставляет комментарий.

## 25. Ограничения первой версии

В MVP не требуется реализовывать:

* видеозвонки;
* чат между сотрудниками;
* диаграмму Ганта;
* учёт отработанных часов;
* интеграцию с Telegram;
* интеграцию с электронной почтой;
* импорт задач из Jira;
* создание пользовательских типов задач;
* сложную систему прав доступа;
* несколько рабочих пространств;
* платную подписку.

Эти возможности могут быть добавлены после тестирования первой версии.

## 26. Критерии приёмки

Приложение считается готовым к приёмке, если:

1. Пользователь может войти в систему и выйти из неё.
2. Администратор может создавать проекты и пользователей.
3. Руководитель может создавать и редактировать задачи.
4. Задачу можно назначить сотруднику.
5. Задачи отображаются списком и на Kanban-доске.
6. Карточки можно перемещать между статусами.
7. Пользователь может оставлять комментарии.
8. Работают фильтрация и поиск.
9. Просроченные задачи визуально выделяются.
10. Уведомления появляются внутри приложения.
11. Интерфейс корректно отображается на компьютере и смартфоне.
12. После обновления страницы сохранённые данные не исчезают.
13. Пользователь не видит проекты, к которым у него нет доступа.
14. В приложении присутствуют демонстрационные данные.
15. Нет критических ошибок, блокирующих основные сценарии.

## 27. Ожидаемый результат

Заказчик должен получить готовое адаптивное веб-приложение, которое можно открыть в браузере, протестировать на компьютере и смартфоне и использовать для демонстрации основных процессов управления задачами в компании NordFlow.
